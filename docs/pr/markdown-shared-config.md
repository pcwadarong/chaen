## 📝 작업 내용

마크다운 렌더링 규칙을 서버 전용 컴포넌트에 인라인으로 묶어두지 않고, 공용 설정으로 승격해 편집 프리뷰와 실제 발행 화면이 같은 규칙을 보도록 정리하는 작업이다.

현재까지 반영한 내용은 다음과 같다.

1. `remark-breaks`를 추가해 단일 줄바꿈도 본문 의도대로 `<br>`로 해석되게 만들었다.
2. 기존 `MarkdownRenderer`에 있던 remark/rehype 설정과 커스텀 컴포넌트 매핑을 공용 설정 모듈로 분리했다.
3. article 본문에 적용되던 markdown wrapper 스타일도 공용 모듈로 함께 옮겨 preview가 동일한 스타일 규칙을 재사용할 수 있게 했다.
4. 서버 렌더링 테스트에 단일 줄바꿈 케이스를 추가해 회귀를 막았다.

<br/>

## 🎯 이번 작업의 핵심

### 1. preview와 production의 규칙 분기를 없애기 위한 기반 정리

편집 화면 preview를 나중에 붙이더라도, 실제 발행 화면과 옵션이 따로 놀면 줄바꿈, 코드 블럭, 링크 처리, 표 렌더링에서 쉽게 차이가 생긴다.

그래서 이번 단계에서는:

- remark 플러그인 구성
- rehype 코드 하이라이트 구성
- 커스텀 컴포넌트 매핑
- 본문 wrapper 스타일

을 공용 모듈 하나로 모아 동일한 입력이 동일한 출력 규칙을 타도록 정리했다.

### 2. 줄바꿈 정책을 markdown 편집 UX에 맞게 조정

기존 설정에서는 단일 줄바꿈이 일반 공백처럼 처리됐다. 이는 기대하는 결과와 실제 발행 결과 사이에 차이를 만든다.

이번 변경으로:

- 단일 줄바꿈은 `<br>`로 렌더링된다.
- 빈 줄은 기존처럼 문단 구분으로 유지된다.
- GFM, 표, 코드 블럭 같은 기존 확장 기능은 그대로 유지한다.

<br/>

## 🚨 판단 근거

### 왜 renderer 내부 리팩터링보다 공용 설정 추출을 먼저 했는가

preview 컴포넌트를 먼저 만들면 결국 서버/클라이언트 구현이 각자 옵션을 들고 있게 된다. 그러면 같은 문제를 다른 위치에 복제하게 된다.

따라서 먼저 공용 설정을 만든 뒤, 이후 preview는 그 설정을 그대로 소비하는 방향이 맞다.

### 왜 스타일도 함께 공용화했는가

preview와 production이 "같은 플러그인"만 공유해도 wrapper 스타일이 다르면 실제 체감 결과는 달라진다.

이번 범위에서는 스타일 회귀 없이 동일한 본문 간격/타이포/테이블/코드 블럭 스타일을 재사용할 수 있도록 wrapper 클래스도 공용화했다.

<br/>

- 단일 줄바꿈 회귀 테스트 추가
